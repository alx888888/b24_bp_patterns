# AGENTS.md

## Назначение проекта

Этот каталог используется как рабочая база примеров для Bitrix24.
Основной артефакт проекта: `patterns.md`.

В проекте агент помогает с такими задачами:

- разработка и уточнение шаблонов для бизнес-процессов Bitrix24;
- подготовка выражений, формул и условий;
- подготовка REST-запросов и JSONPath для `REST Активити Б24`;
- оформление и систематизация примеров по темам;
- сокращение дублей и поддержание читаемости справочника;
- фиксация рабочих решений для CRM, товаров, каталога, сообщений, BBCode и регулярных выражений.


## Основной файл

Главный файл проекта:

- `patterns.md`

Если пользователь прямо просит добавить новый практический пример реализации по Bitrix24, пример должен попадать именно в `patterns.md`, если пользователь не указал другой файл.

Распределение материалов по проекту:

- `patterns.md` — короткие конкретные примеры реализации: формулы, шаблоны, REST-запросы, JSONPath, BBCode, регулярные выражения.
- `docs/` — новые кейсы, разборы, заметки и расширенные практические сценарии, которые не подходят по формату для `patterns.md`.
- `docs/rest-activity/` — только локальная база документации, загруженной по приложению `REST Активити Б24`.


## Роль агента в этом проекте

Агент в этом проекте действует как редактор и разработчик прикладных шаблонов Bitrix24.
Без прямого запроса пользователя агент не добавляет новые примеры в `patterns.md` и не меняет существующие шаблоны по собственной инициативе.

Если задача касается разработки, настройки или применения `REST Активити Б24`, сначала смотри локальную базу знаний в `docs/rest-activity/`.
Точка входа: `docs/rest-activity/README.md`.
Статьи из этой папки используй как справочник по возможностям приложения, примерам запросов и типовым сценариям.
Новые пользовательские кейсы, заметки и разборы в `docs/rest-activity/` не добавляй.
Такие материалы сохраняй в `docs/`, если пользователь просит сохранить именно кейс или расширенное описание.

Приоритеты работы:

1. Дать рабочее решение под реальный кейс.
2. Сохранить единый стиль файла `patterns.md`.
3. Не плодить дубли, если пример уже существует.
4. Не усложнять шаблон, если задача решается проще.
5. Держать файл в формате удобного справочника, а не лога переписки.


## Правила оформления `patterns.md`

### 1. Структура разделов

Файл должен быть разбит на крупные тематические разделы первого уровня:

- `# REST и вебхуки`
- `# Сборки и готовые шаблоны`
- `# Даты и время`
- `# Числа, деньги и математика`
- `# Строки и извлечение данных`
- `# Массивы`
- `# Условия и логика`
- `# Примеры сообщений для мессенджеров`
- `# BBCode`
- `# Регулярные выражения`
- `# Шаблоны печатных форм документов`
- `# Справка и заметки`

Новые примеры должны попадать в подходящий тематический раздел.


### 2. Заголовки примеров

Каждый самостоятельный пример внутри раздела оформляется как заголовок четвертого уровня:

```md
#### Название примера:
```

Примеры:

```md
#### Отложить заявку на N дней к 08:00:
#### Получить список себестоимостей товаров:
#### Поле типа Деньги вида 1000|RUB преобразовать только в число:
```


### 3. Внутренние подписи внутри одного примера

Если у примера есть внутренние подписи, они оформляются обычным текстом, без решеток:

```md
Метод:
Запрос:
JSONPath:
Синтаксис:
```

Правильный формат:

````md
#### Получить список себестоимостей товаров:

Метод:
`catalog.product.list`

Запрос:
```json
{
  "select": ["id", "iblockId", "name", "purchasingPrice"]
}
```

JSONPath:
`$.products.*.purchasingPrice`
````

Неправильный формат:

```md
#### Метод:
#### Запрос:
#### JSONPath:
```


### 4. Формат самих примеров

Оформление зависит от типа материала:

- одиночная формула: одной строкой;
- несколько связанных формул: компактным блоком;
- REST-запрос: с подписями `Метод:`, `Запрос:`, `JSONPath:`;
- JSON и код: в fenced code block;
- готовые URL-шаблоны: отдельной строкой;
- пояснение к примеру: короткое, прикладное, без лишней теории.


### 5. Принципы наполнения

- Добавлять только практически полезные и проверенные примеры.
- Если пример уже есть, не создавать дубль без необходимости.
- Если новый пример точнее старого, обновлять существующий блок.
- Примеры для `REST Активити Б24` держать в разделе `REST и вебхуки`.
- Короткие рабочие примеры реализации держать в `patterns.md`.
- Расширенные кейсы и разборы держать в `docs/`.
- Папку `docs/rest-activity/` не использовать для новых авторских кейсов проекта.
- Для облачного Битрикс24 не предлагать `replace()` как универсальный прием без проверки: в облаке функция может не отрабатывать.
- Для удаления префиксов и фиксированных частей строки в первую очередь использовать `substr()` и `strpos()`.
- Примеры сообщений не смешивать с BBCode и REST.
- Регулярные выражения держать отдельным разделом.
- BBCode держать отдельным разделом.


### 6. Работа с существующим содержимым

- Не удалять рабочие примеры без явной причины.
- Не распылять одинаковые по смыслу примеры по разным разделам.
- Если пример спорный, лучше уточнить подпись, чем переписывать его без нужды.
- При реорганизации файла сохранять содержимое, если пользователь явно не попросил удалить его.


### 7. Безопасность

В файле нельзя хранить:

- боевые `apikey`;
- webhook-токены;
- приватные ключи;
- чувствительные URL с доступом к реальным данным или действиям.

Если нужно сохранить форму примера, секреты заменяются заглушками:

- `YOUR_API_KEY`
- `YOUR_WEBHOOK`
- `YOUR_DOMAIN`


## Как агенту вносить изменения

Если пользователь прямо просит добавить новый пример, агент должен:

1. Найти подходящий тематический раздел.
2. Проверить, нет ли уже такого же или очень похожего примера.
3. Выбрать правильное место хранения:
   - `patterns.md` для короткого рабочего примера реализации;
   - `docs/` для кейса, разбора или расширенного описания;
   - не использовать `docs/rest-activity/` для новых кейсов проекта.
4. Не менять без необходимости другие рабочие блоки.
5. Если изменяется форматирование файла, делать его более последовательным, а не просто другим.
6. После запроса пользователя на добавление нового шаблона в базу зафиксировать изменения в `git`: проверить `git diff`, выполнить `git add` для затронутых файлов и создать коммит с коротким сообщением по сути правки.
7. Если пользователь просит обновить удалённый репозиторий, отправить коммит на GitHub через `git push`. Pull request выполнять только по отдельному запросу пользователя.


## Что считается хорошим результатом

Хороший результат в этом проекте:

- шаблон реально работает в Bitrix24;
- пример лежит в правильном разделе;
- подпись понятна с первого взгляда;
- нет лишнего шума;
- файл остаётся удобным как практический справочник.
